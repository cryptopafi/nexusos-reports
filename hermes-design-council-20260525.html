Summary

All three advisors support accepting hermes-design as the correct standalone direction, but only while it remains disabled and draft-only. The shared rationale is strong isolation, explicit denied actions, and activation gates. The remaining risk is not architecture direction; it is enforcement quality before activation.

Agreement Zones

Standalone profile isolation is the right architecture. Response A says isolation matches Delphi D3 and avoids contaminating production with experimental creative tooling, Response B says it prevents prompt dilution and limits blast radius from Hermes Main, and Response C says it reduces exposure across pilot files, provider credentials, publishing workflows, and production systems.

Disabled/draft-only posture is adequate for the current installation phase. Response A cites denied publishing, payments, secret reads, and pilot writes; Response B highlights disabled Gateway and secret wiring barriers; Response C cites disabled gateway, disabled credentials, no live secrets, denied actions, and audit gating.

Activation must remain blocked until provider routing, budget policy, credentials, compliance, brand kit, E2E tests, and /audit-pro PASS are complete. Response A lists these gates directly, Response B emphasizes zero CRITICAL/HIGH /audit-pro and compliance checklists, and Response C repeats the need for provider routing, budget policy, service-account activation, approved brand kit, and real-provider E2E tests.
