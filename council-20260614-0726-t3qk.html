Summary

All three advisors recommend PASS for a hybrid shared-core approach: keep Hermes PA and Hermes Concierge separate for memory, audit, schema, and workflow state, while giving Pafi one natural conversational front door and reusing common safety/approval infrastructure. The main dissent is not about the recommendation; it is about implementation strictness: Response C is more explicit that the shared core should be a narrow safety/approval/audit kernel with deterministic routing labels and parity tests, while Responses A and B emphasize a unified user experience and internal routing.

Substantive Answer

Recommendation: implement Hermes Concierge as a hybrid shared-core extension with separate surfaces.

Plain-English explanation: Pafi should be able to talk naturally to Hermes without choosing a bot. Hermes PA remains the everyday private assistant for inbox, calendar, reminders, personal drafts, Google Workspace reads, Telegram PA gateway, and approval packets. Hermes Concierge remains a separate disabled profile for external coordination workflows such as travel, vendors, services, booking plans, option packets, and approval-gated external action preparation. Under the hood, PA and Concierge keep separate memory, audit, namespaces, and schemas. Shared code should cover common safety checks, redaction, approval-packet generation, audit mechanics, and draft formatting, but shared code must not mean shared memory.

What belongs in Hermes PA:

  • Inbox triage, reply drafts, prioritization, reminders, schedule reasoning, calendar support, and personal task tracking.
  • Owner-facing Google Workspace reads and source-link front-door handling, with canonical intelligence delegated elsewhere when appropriate.
  • Telegram PA gateway and simple external-contact drafts that do not require a structured vendor, travel, booking, or service workflow.
  • Sensitive personal preference memory, isolated and approval-gated.

What belongs in Hermes Concierge:

  • Travel planning, itineraries, hotel/flight/restaurant/transport comparisons, vendor research, service planning, booking plans, cancellation plans, and option packets.
  • Vendor/service coordination drafts and external logistics workflows.
  • Workflow state for travel, vendors, services, and external coordination, isolated from personal PA state.

Namespaces and schemas should be preserved:

  • Hermes PA: tenant/nexusos/profile/hermes-pa, voxsolutions/pod/personal-assistant, voxsolutions/audit/personal-assistant, pa_task_pack_v1.
  • Hermes Concierge: tenant/nexusos/profile/hermes-concierge, voxsolutions/pod/concierge, voxsolutions/audit/concierge, concierge_task_pack_v1.

Routing model: PA should be the normal front door. Route internally to Concierge when the request involves travel, vendors, services, bookings, cancellations, multi-option external comparisons, logistics, or external coordination packets. Ambiguous requests should default to PA unless Concierge workflow signals are strong. Mixed requests should be split into PA and Concierge parts. The output should label the active surface clearly when Concierge is involved, for example [Concierge draft], so Pafi gets natural conversation without hidden state confusion.

Required approval boundaries: no profile may execute sensitive disclosure, external sends, public posts, broadcasts, bookings, payments, cancellations, vendor commitments, browser form submissions, identity use, credential reads, credential operations, or secret access without explicit scoped approval. Calendar holds and external scheduling actions also need their relevant approval gate. Allowed without approval: internal analysis, drafts, option comparisons, approval packets, risk labels, and asking Pafi for missing details.

Activation plan:

1. Keep Hermes Concierge disabled.

2. Extract or confirm a shared safety/approval/audit kernel only where behavior is genuinely common.

3. Run a PA parity audit to verify PA behavior does not regress.

4. Audit Concierge namespaces, schema, tool permissions, memory boundaries, audit logging, cron state, shell access, credential access, external-send access, booking access, payment access, and browser-submit access.

5. Add routing rules for travel, vendor, service, booking, cancellation, and option-packet workflows.

6. Add tests for routing, approval blocking, memory isolation, schema output, and audit log placement.

7. Run Concierge in shadow/draft mode on simulated workflows.

8. Require operator activation after audit pass.

9. Enable only draft/plan/read behavior first; keep all external actions approval-gated and never auto-executed.

Risks and mitigations:

  • PA context leaks into Concierge: default-deny memory transfer; allow only sanitized, scoped, approved facts.
  • User confusion: keep PA as front door, label Concierge outputs, and split mixed workflows.
  • PA bloat: keep travel/vendor/service workflow logic in Concierge.
  • Profile sprawl: keep only one Concierge surface and deduplicate shared utilities.
  • Shared-kernel regression: version the kernel and require PA parity plus cross-surface approval conformance tests.
  • Premature activation: Concierge remains disabled until audit and operator gates pass.

Kill criteria:

  • Disable Concierge immediately if any external send, booking, payment, cancellation, commitment, browser submit, identity use, or credential operation occurs without a matching approval record.
  • Disable Concierge if PA private memory appears in Concierge workflow state or in an external-facing draft without explicit approved transfer.
  • Roll back the shared kernel if PA parity tests show behavioral regression.
  • Merge back into PA if Concierge proves to be only a few templates with no meaningful workflow state and creates routing confusion.
  • Strengthen separation or split fully if the shared core causes repeated cross-surface coupling bugs or the safety contracts diverge.
  • Keep Concierge disabled if audit finds active external-send, booking, payment, browser-submit, credential, or identity-use capability without approval gates.

Assumptions and evidence gaps: the advisors did not inspect live profile YAML or runtime router code. The router, shared-kernel extraction feasibility, and parity behavior are implementation assumptions that must be verified during audit.

Final Synthesis Trace

The final verdict is PASS because all three advisors independently converge on the same architecture: hybrid shared-core with separate PA and Concierge surfaces. The strongest shared claim is that this is the only option that simultaneously minimizes Pafi-facing complexity, preserves memory and audit isolation, avoids PA bloat, avoids unnecessary profile sprawl, and keeps all external-risk actions approval-gated.

The dissent affects implementation, not the top-level decision. Response C's stricter routing labels, kernel boundaries, parity testing, and conformance testing should be incorporated because they directly reduce the highest-stakes risks identified in the brief: sensitive context leakage, premature activation, unsafe external commitments, and hidden routing confusion. Response A's and Response B's user-experience concern should also be preserved: Pafi should not need to manually pick profiles during ordinary conversation.

Therefore, the reconciled answer is: build the hybrid; use PA as the front door; preserve Concierge as a disabled separate surface; share only common safety/approval/audit utilities; label Concierge workflows; require scoped approvals for all external-risk actions; and activate only after audit, isolation, routing, parity, and operator gates pass.
