FORGEBUILD Portable Skill — Council Reconciliation

Verdict

INTEGRATE_IN_FORGEBUILD (unanimous; confidence 0.82 / 0.88 / 0.95)

Recommendation

Add PORTABLE-SKILL as a first-class fourth path inside FORGEBUILD (alongside CREATE / CONVERT / ASSEMBLE). User-facing surface is forgebuild portable-skill with subcommands init, lint, build, install. Any skillforge artifact stays as an internal builder script/library, not a separate user-facing skill.

Rationale

  • - Cohesion with existing scope. FORGEBUILD already owns create/convert/assemble for procedures, skills, and plugin bundles, plus wrappers, validation, and portability concerns — multi-runtime compilation is the natural next path, not a new domain.
  • - Lower agent cognitive load. A second top-level skill (skillforge) creates routing ambiguity ("which one converts? which one builds multi-runtime?") — exactly the failure mode that produced the /opt Claude/Codex frontmatter incident.
  • - Reuse over duplication. Skill search, component reuse, wrapper generation, and validation already live in FORGEBUILD; a sibling skill would drift, while a sub-protocol composes cleanly with CONVERT (single-runtime → portable) and ASSEMBLE (portable → plugin bundle).
