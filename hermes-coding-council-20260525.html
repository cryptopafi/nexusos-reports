Summary

The council broadly supports the disabled-first Hermes Coding direction, but the installation plan should be revised before disabled install. The strongest shared signal is that disabled-first staging, OpenClaw fallback, provider-router enforcement, denied-action tests, and Hermes Pilot isolation are strategically sound. The decisive revision need is sharper implementation evidence: pinned model routes, post-execution verification against fabricated success, explicit Pilot protected paths, workload-class parity fixtures, and rollback/promotion criteria.

Agreement Zones

Disabled-first staging is the right strategic posture. Response A says the verification-key gate prevents premature production activation, Response B says live coding jobs are blocked until Stage 1 E2E evidence exists, and Response C says Stage 1 gates limit initial blast radius.

Adapting NexusOS TECH patterns is directionally sound while OpenClaw remains fallback. Response A highlights proven NexusOS multi-agent patterns and Cortex prior art, while Response B and Response C both treat OpenClaw hardening lessons as incorporated into the plan through routing, denied actions, and gate requirements.

Provider-router middleware is a non-negotiable control. Response A emphasizes closing bare model names, silent fallback, write bypass, and weak pinning; Response B calls provider-router coverage a critical dependency; Response C warns that provider-router failure could bypass policy across Codex, Claude Code, and NexusOS TECH tiers.

Hermes Pilot isolation is essential. Response B flags the unspecified protected Pilot paths as high risk, and Response C flags local file path isolation risks including symlink and path traversal bypasses. Response A's concern about fabricated success also applies to Pilot-untouched evidence unless the audit is schema-backed and independently verified.

Workload-class parity must precede OpenClaw replacement. Response A says promotion criteria and rollback triggers are under-specified, Response B says workload-class parity needs acceptance fixtures and regression history, and Response C supports measurable success criteria before any expansion.
