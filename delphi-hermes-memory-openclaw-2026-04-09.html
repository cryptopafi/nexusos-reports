Executive Summary

Hermes v0.8.0 memory uses a frozen-snapshot pattern: MEMORY.md and USER.md are loaded into the system prompt once at session start and never updated mid-session, preserving the Anthropic prefix cache. The system prompt itself is cached in SQLite (state.db) and reused across gateway restarts for each session, with a SOUL.md mtime check as the only rebuild trigger.


OpenClaw's HTTP API (port 18789) is UI-only — there is no documented REST API for programmatic agent interaction by default. The correct interconnection path between Hermes and OpenClaw is via Hermes's own OpenAI-compatible API server (port 8642, api_server platform), not through OpenClaw's port.


Critical: Four known memory bugs exist in Hermes (issues #2670, #3059, #4708, #5129) that affect gateway-mode deployments. Two are HIGH RISK for Lis.
Dead file: lessons.md at ~/.hermes/lessons.md is never loaded by any Hermes Python code. The 49 rules seeded there are NOT reaching Lis.

System Prompt Slot Order (Exact Assembly)

Source: run_agent.py:2657-2820, prompt_builder.py

The _build_system_prompt() method assembles slots in this exact order:

  1. SOUL.md — from ~/.hermes/SOUL.md via load_soul_md(). Falls back to DEFAULT_AGENT_IDENTITY if not found.
  2. Tool guidance block — appended only if tools are loaded: MEMORY_GUIDANCE, SESSION_SEARCH_GUIDANCE, SKILLS_GUIDANCE.
  3. Nous subscription prompt — only if Nous subscription tools are active.
  4. Tool-use enforcement guidance — injected for GPT/Codex/Gemini/Gemma/Grok models.
  5. system_message — gateway-injected system prompt from config.yaml gateway.system_prompt.
  6. MEMORY.md frozen snapshot — if memory_enabled: true in config.
  7. USER.md frozen snapshot — if user_profile_enabled: true.
  8. External memory provider block — if Honcho, Mem0 etc. configured.
  9. Skills index — list of available skills from ~/.hermes/skills/.
  10. Context files — first match wins: .hermes.md → AGENTS.md → CLAUDE.md → .cursorrules.
  11. Timestamp + session ID + model/provider — frozen at build time.
  12. Platform hint — e.g. Telegram formatting hints.
Key detail: lessons.md is referenced in MEMORY.md content as a path but is NOT loaded by any Python code. It is a dead file. Lessons must be placed in SOUL.md (slot #1) or referenced as a .hermes.md context file (slot #10).

System Prompt Caching — What Triggers a Rebuild

Source: run_agent.py:7130-7200

The _cached_system_prompt is cached per-session in two places: in-memory (self._cached_system_prompt) and SQLite (state.db).

What triggers a rebuild

  • SOUL.md mtime check — if os.path.getmtime(SOUL.md) > session_row.get("started_at"), cache is invalidated. This is the primary mechanism.
  • Context compression events — set self._cached_system_prompt = None explicitly (line 1396).
  • New session — no stored session in DB means fresh build.
  • Model/provider swap — _reinit_client() invalidates the cache (line 1395-1396).

What does NOT trigger rebuild

  • Writing to MEMORY.md/USER.md mid-session (by design — frozen snapshot pattern)
  • Plugin state changes
  • Skills changes mid-session
Practical implication for Lis: Updating SOUL.md triggers rebuild on next message. Updating MEMORY.md manually takes effect at next session start OR after compression.

Memory Write-Back — How Writes Actually Happen

Source: memory_tool.py, run_agent.py:1005-1095, 5850-5930

A. Direct Tool Call (memory tool)

When the model calls the memory tool with action=add/replace/remove, MemoryStore.save_to_disk() is called immediately. Uses atomic rename via tempfile + os.replace() for safety. File-locked with fcntl.LOCK_EX on a .lock sidecar file.

B. Background Memory Review (nudge system)

After every N turns (nudge_interval, default 10), Hermes spawns a background AIAgent clone that reviews the conversation and decides what to save.

  • Gets a forked snapshot of messages
  • Has _memory_nudge_interval = 0 and _skill_nudge_interval = 0 (prevents recursive nudges)
  • Writes directly to the shared MemoryStore

Config parameters (VPS config.yaml)

ParameterYour VPS ValueDefaultDescription
nudge_interval1010Background review every N user turns
flush_min_turns66Min turns before flush triggers on compression
memory_char_limit40002200Max chars in MEMORY.md
user_char_limit20001375Max chars in USER.md

C. Flush on Session End/Compression

flush_memories() is called before context compression or session reset. It injects a special compression system message, makes a single API call with only the memory tool available, then strips all flush artifacts. Requires user_turn_count >= flush_min_turns.

Memory Persistence Across Gateway Restarts

Source: gateway/session.py, run_agent.py

What persists (survives restarts)

  • ~/.hermes/memories/MEMORY.md — file-backed, survives everything
  • ~/.hermes/memories/USER.md — file-backed, survives everything
  • ~/.hermes/state.db — SQLite WAL-mode: session metadata, message transcripts (FTS5), stored system prompts, pre-flushed session flags
  • ~/.hermes/sessions/ — JSONL transcript files (state.db backup)

What is lost on restart

  • _pre_flushed_sessions in-memory set — root cause of Bug #3059 (now fixed in recent versions)
  • In-flight background review threads — safe to lose, no data loss risk
  • _cached_system_prompt in-memory — rebuilt from state.db on next message
Session isolation: Each Telegram user gets their own session key (built from platform + user_id + chat_id + thread_id). Conversation history and memory context are isolated per user.

Known Memory Bugs — Priority Order

Source: GitHub issues NousResearch/hermes-agent

BUG-1: Double Memory Flush on Restart High Risk — Lis

Issue #3059. When a session expires and triggers memory flush, if the gateway restarts, the same session flushes again. The _pre_flushed_sessions set was in-memory only. Status: Fix landed (persisting flag to state.db). Verify your version has this fix.

BUG-2: Memory Flush Overwrites Live Memory High Risk

Issue #2670. The flush agent has no awareness of memory changes made during the current session. It can revert memory to stale state on session timeout. Mitigation: Keep flush_min_turns: 6 and save critical facts via tool calls before ending long sessions.

BUG-3: Background Review Duplicate Provider Medium Risk

Issue #5129. Background review spawns a clone that creates a second instance of the memory provider. Can create duplicate memory entries. Mitigation: SQLite WAL prevents corruption; duplicates deduplicated via dict.fromkeys() on load.

BUG-4: Memory Provider Tool Routing Failure Low Risk

Issue #4708. Affects external memory providers (Honcho, Mem0). fact_store, fact_feedback tools return Unknown tool error in sequential dispatch path. Impact on Lis: None — using built-in MEMORY.md/USER.md, no external provider configured.

  • Auth token: 20d352bacd...a844df (from openclaw.json), uses Bearer token auth

To enable programmatic HTTP access

{
  "gateway": {
    "http": {
      "endpoints": {
        "chatCompletions": { "enabled": true }
      }
    }
  }
}

Then POST to http://127.0.0.1:18789/v1/chat/completions with Authorization: Bearer <token>. Use model: "openclaw:<agent-name>" to target specific agents.

OpenClaw Agent Routing Details

Source: .openclaw/openclaw.json, docs.openclaw.ai/tools/agent-send, container inspection

Native agent communication

OpenClaw has a native agent_send tool for dispatching tasks between agents. The A2A v0.3.0 protocol plugin (openclaw-a2a-gateway) enables cross-server agent communication.

Hermes to OpenClaw routing (step by step)

  1. Enable gateway.http.endpoints.chatCompletions.enabled: true in OpenClaw config
  2. POST to http://127.0.0.1:18789/v1/chat/completions
  3. Set Authorization: Bearer 20d352bacd15538460c654ba1a864025114c773f82a844df
  4. Set model: "openclaw:<agent-name>" to target a specific agent

Note: The 6 business agents (Richard, Finance, Marketing etc.) are configured on MacIntel, not VPS.

Community Best Practices for Hermes Memory

Source: hermes-agent.nousresearch.com/docs, vectorize.io, deepwiki.com/NousResearch/hermes-agent/4.3

Save to MEMORY.md

  • Environment facts (VPS IPs, paths, service ports)
  • Tool quirks and workarounds discovered
  • Project conventions (naming, structure)
  • Recurring corrections
  • "What reduces future user steering"

Do NOT save

  • Task progress or session outcomes
  • Completed-work logs
  • Temporary TODO state
  • Info easily re-discovered from context files

Memory vs Skills distinction

  • Memory = "what" (facts, preferences, env details)
  • Skills = "how" (multi-step workflows, reusable recipes)

Char limit strategy (your config: 4000 chars)

  • 4000 chars ≈ 25-35 medium-sized entries
  • When full: use replace action (not remove + add) to consolidate
  • GROUP related facts into single entries to maximize density
