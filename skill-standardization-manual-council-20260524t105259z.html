Skill Standardization Council — Reconciled Decision

Verdict

STRONG_PASS (consensus across all three advisors; confidence range 0.82–0.95). Adopt the three-layer portable skill architecture as the canonical standard.

Recommended Architecture

Canonical source tree at skills-src/<skill>/ containing:

  • - SKILL.core.md — single behavioral source of truth (runtime-neutral prompt/workflow)
  • - targets/<runtime>/frontmatter.yaml — the only place for model, tool, path, env, permission, reasoning-effort declarations
  • - targets/<runtime>/wrapper.md or run.sh — optional execution adapter

A skillforge build (a.k.a. portable-skill build) step composes these into immutable build/<runtime>/SKILL.md artifacts with embedded SHA256 hashes of core + target. Only build/ outputs are ever synced into runtime skill directories (~/.claude/skills/, Codex skill dir, Hermes/OpenClaw paths). Generated installs carry a hash header and are treated as build artifacts, never hand-edited.

Wrapper Policy

Mandatory when a skill declares or touches any of: model, reasoning_effort, tool name, command alias, filesystem path, permission, environment variable, secret, state/output directory, MCP/CLI call, runtime-specific failure behavior, or targets a runtime with a non-Markdown native format (e.g. OpenClaw's openclaw.plugin.json).

Optional only for pure-prompt/process skills whose core is fully runtime-neutral and whose frontmatter contains no executable or provider-specific fields. Even when optional, an empty wrapper stub is permitted so the build path stays uniform.

Wrappers may only map runtime-specific execution details — they must not restate or reimplement core workflow logic (enforced by lint).
